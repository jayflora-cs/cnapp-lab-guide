CNAPP Workshop Lab Repository

A complete, automated lab environment for demonstrating CrowdStrike Falcon Cloud Security capabilities. This guide explains every script, file, and component — and how they work together to deliver an end-to-end cloud security workshop.

The Big Picture

This repository deploys a two-VM environment on AWS that demonstrates the full CrowdStrike CNAPP story:

Victim VM (Ubuntu)

Runs K3s (lightweight Kubernetes) with a CI/CD pipeline (Gitea + ArgoCD), a deliberately vulnerable Flask app ("Peregrine"), and CrowdStrike Falcon agents.

Attacker VM (Windows)

Pre-configured Windows desktop with Firefox and Metasploit. Participants RDP in and launch guided attack simulations against the victim.

The workshop flow takes participants through:

Deploy Code → CI/CD Scan → Fix Vulns → GitOps Deploy → KAC Admission → Attack Sim → Falcon Detection

deploy.sh

Script The main orchestrator. Runs the entire deployment from scratch with a single command.

What it does (in order):

1. Preflight checks — verifies terraform, ansible, and aws CLI are installed
2. Detects AWS profile from terraform.tfvars
3. Runs terraform init + terraform apply -auto-approve
4. Extracts victim IP, attacker IP, and environment alias from Terraform outputs
5. Waits for SSH to become available on the victim VM (up to 5 min)
6. Runs Ansible playbook (victim.yml) over SSH
7. Prints a summary with all service URLs and credentials

destroy.sh

Script One-command teardown. Destroys all AWS resources.

What it does:

Detects the AWS profile from terraform.tfvars, then runs terraform destroy -auto-approve. All EC2 instances, security groups, EIPs, and IAM resources are removed.

Terraform — Infrastructure

Terraform The terraform/ directory defines all AWS infrastructure as code.

Core Files

FilePurpose
provider.tfConfigures AWS provider (~5.0), TLS, local, and random providers. Sets region/profile and applies default project tags.
variables.tfAll input variables: AWS region/profile, instance types, Falcon credentials, lab_mode, lab_scope, attack_mode, allowed_ips.
terraform.tfvars.exampleExample config — copy to terraform.tfvars and fill in your values.
network.tfReferences the default VPC and subnets (avoids VPC limit issues).
security.tfTwo security groups: victim (restricts inbound to allowed_ips) and attacker (allows RDP). Cross-SG rules for attacker↔victim traffic.
compute.tfRSA key pair, Ubuntu 22.04 AMI lookup, victim EC2 + Elastic IP, Windows Server 2022 AMI for attacker, IAM instance profile, SSM provisioning.
outputs.tfExports IPs, SSH command, service URLs, RDP details. Generates the Ansible inventory file via templatefile().

Ansible — Provisioning

Ansible The ansible/ directory installs and configures all software on the victim VM via SSH.

Playbooks

PlaybookUse Case
playbooks/victim.ymlStandard — runs over SSH from your laptop. Applies roles in order.
playbooks/victim-local.ymlCloudShare variant — runs with connection: local on the VM itself.
playbooks/attacker.ymlDeprecated — Windows attacker is now provisioned via PowerShell user_data.

Role Execution Order

common → k3s → gitea → falcon → argocd → sample-app → auto-attack → lab-guide

Role: common

ansible/roles/common/tasks/main.yml

Installs base system packages (curl, wget, git, jq), Docker engine, Helm CLI, and kernel sysctl settings. Intentionally installs liblog4j2-java to enable a Log4Shell (CVE-2021-44228) detection demo scenario.

Role: k3s

ansible/roles/k3s/tasks/main.yml

Installs K3s v1.30.2 (single-node, traefik disabled). Configures the private container registry to point at Gitea. Sets up kubeconfig for the ubuntu user. Waits for the cluster to become ready before continuing.

Role: gitea

ansible/roles/gitea/tasks/main.yml

Adds Gitea Helm repo, installs Gitea via Helm chart with custom values (NodePort 30300, admin user, Actions runner). Creates the workshop admin user. Deploys a Gitea Actions runner pod that executes CI/CD pipelines.

Templates:

gitea-values.yml.j2Helm values: NodePort, credentials, persistence, registry config
gitea-runner.yml.j2K8s Deployment for the CI/CD Actions runner

Role: falcon

ansible/roles/falcon/tasks/main.yml

Conditionally deploys CrowdStrike Falcon (when auto_deploy_falcon=true). Auto-detects the cloud region via OAuth2 probe. Auto-detects the CID. Deploys the falcon-platform umbrella Helm chart (sensor + KAC + IAR). Injects Falcon API secrets into Gitea as Actions secrets for CI/CD image scanning.

Role: argocd

ansible/roles/argocd/tasks/main.yml

Installs ArgoCD v3.3.6 via Helm. Applies a copyutil patch (fixes CrashLoopBackOff). Reduces CPU requests for lab sizing. Sets the admin password. Registers the Gitea repository and creates the ArgoCD Application for the Peregrine app.

Role: sample-app

ansible/roles/sample-app/tasks/main.yml

Copies the Peregrine application files, initializes a Git repo on Gitea, pushes all code (including the CI/CD pipeline definition), triggers the first CI/CD build, and waits for ArgoCD to sync the deployment to K3s.

Key files pushed to Gitea:

app.pyFlask app with intentional command injection at /api/debug/health-check
requirements.txtIncludes aiohttp 3.9.1 (known CVEs for detection)
DockerfilePython 3.12 image with iputils-ping for the attack demo
deployment.yamlrunAsUser: 0 and hostPath mount — intentionally blocked by KAC
build-deploy.yamlGitea Actions CI/CD workflow with FCS image scanning

Role: auto-attack

ansible/roles/auto-attack/tasks/main.yml

Controls attack delivery based on attack_mode:
• hands_on — skips (participants use Metasploit manually)
• button — stages a K8s Job manifest for one-click triggering
• auto — stages + immediately launches the attack Job at provision time

Role: lab-guide

ansible/roles/lab-guide/tasks/main.yml

Fetches a malicious domain from Falcon Intel API (for IOC demo). Builds the React lab-guide Docker image. Pushes to Gitea's container registry. Deploys to K3s with a ConfigMap containing runtime config (IPs, CID, lab mode, scope, attack mode).

CI/CD Pipeline

The Gitea Actions workflow (.gitea/workflows/build-deploy.yaml) runs on every push to main:

Docker Build → FCS Image Scan → Push to Registry → Update K8s Manifest → ArgoCD Sync → KAC Check

The pipeline intentionally fails on the first run due to CVE-2020-14343 in PyYAML. Participants learn to fix the vulnerability and re-run.

Two Modes

Walkthrough Mode

17 guided steps (~90 min). Step-by-step instructions with code snippets, screenshots, and checklists.

Challenge Mode

15 gamified steps (~90-120 min). Scoring, achievements, hints with point penalties, difficulty multipliers.

Runtime Configuration

The app fetches /config.json at startup (mounted as a K8s ConfigMap). This injects environment-specific values like victim IP, Falcon CID, lab mode, lab scope, and attack mode — so the same build works across different deployments.

CloudShare Integration

Platform The cloudshare/ directory enables deployment on CrowdStrike's internal CloudShare/Encounter lab platform (no direct AWS access needed by participants).

FilePurpose
build-stage.shBuilds a tarball of Ansible playbooks (~4 MB) and generates the attacker userdata script. Run before every CloudShare upload.
creation.tfCloudShare-specific Terraform: custom VPC, security groups, EC2 instances, IAM role with scoped attack permissions.
victim/userdata.shVM user-data: downloads the staged tarball, writes vars.yml from CloudShare template variables, runs Ansible locally.
victim/cloudshare-config.jsonCloudShare blueprint config: static vars (Falcon CID, S3 URL) and runtime parameter definitions.
attacker/userdata.ps1Generated file (never hand-edit). Output of build-stage.sh.

Both deployment modes (direct AWS and CloudShare) run the same Ansible roles and produce an identical lab experience. The only difference is who creates the VMs and how variables are injected.

Replicating This Lab

To set up this lab in your own AWS account:

Step 1: Get the prerequisites

# macOS
brew install terraform ansible awscli

# Verify
terraform --version    # >= 1.5
ansible --version      # >= 2.15
aws sts get-caller-identity  # Must return your account

Step 2: Configure your environment

cp terraform/terraform.tfvars.example terraform/terraform.tfvars

Edit terraform/terraform.tfvars:

aws_region           = "us-west-2"           # Your preferred region
allowed_ips          = ["YOUR.IP.HERE/32"]    # Your public IP
victim_instance_type = "t3.large"            # Minimum for K3s + all services

# Optional: auto-deploy CrowdStrike Falcon
auto_deploy_falcon   = true
falcon_client_id     = "your-oauth2-client-id"
falcon_client_secret = "your-oauth2-client-secret"

# Optional: customize lab experience
lab_mode             = "walkthrough"          # or "challenge"
lab_scope            = "full"                 # or "cicd_only", "cdr_only", "express"
attack_mode          = "hands_on"            # or "button", "auto"

Step 3: Deploy

./deploy.sh
# Takes ~8-12 minutes
# Windows attacker VM needs ~10 more minutes after deploy

Step 4: Access

The deploy script prints all URLs. If behind a VPN, use SSH port forwarding (see Quick Start above).

Step 5: Tear Down

./destroy.sh

Cost estimate: ~$0.15/hour (t3.large + t3.medium + EIPs). Always run ./destroy.sh when done to avoid ongoing charges.

Customization

Change the lab experience

SettingOptionsEffect
lab_modewalkthrough / challengeGuided steps vs. gamified with scoring
lab_scopefull / cicd_only / cdr_only / expressWhich exercises are included
attack_modehands_on / button / autoHow the attack simulation is delivered

Add more allowed IPs

# terraform/terraform.tfvars
allowed_ips = ["1.2.3.4/32", "5.6.7.8/32", "9.10.11.12/32"]

# Apply changes
cd terraform && terraform apply

Modify the sample app

Edit files in ansible/roles/sample-app/files/ and ansible/roles/sample-app/templates/, then re-run:

cd ansible && ansible-playbook playbooks/victim.yml -v --tags sample-app

Change instance sizes

Update victim_instance_type / attacker_instance_type in terraform.tfvars. Minimum victim size is t3.large (K3s + all services need ~6GB RAM).
